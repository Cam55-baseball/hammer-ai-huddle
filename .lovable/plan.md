# Plan: Hammers Today Speed Engine — Full Library Integration

## Current state

The Hammers Today speed card is generated by `supabase/functions/wk-generate-daily/index.ts`. Today it uses only the 3 hardcoded slugs returned by `sprintSlugs(sport)` (`accel_10_30y`, `lateral_first_step`/`slap_runner_crossover`, `repeat_90ft_bb`/`repeat_43ft_sb`). The backend has an 80-movement `wk_movement_catalog` speed library with `speed_category` values (`acceleration`, `top_speed`, `reactive`, `plyometric`, `mobility`, `resisted`, `change_of_direction`, `overspeed`) plus Phase 9 templates (`speed.acceleration`, `speed.top_speed`, `speed.mixed`, `speed.elastic`, `speed.game_day_primer`, `speed.practice_day`, `speed.recovery`, `speed.return_to_run`), but the generator is not selecting from that library. This is the bug to fix.

## Goal

Make Hammers Today speed prescriptions fully library-driven, selecting the right speed categories for each phase/day type, using the entire catalog with rotation, preference, and CNS/PAP budgeting, while passing the existing `certifySpeed` validator.

## What I will build

### 1. Create a canonical speed engine module

File: `supabase/functions/_shared/wic/engines/speed.ts`

Exports:
- `SPEED_PREFERRED`: ordered list of elite speed slugs used to break ties when multiple movements fit the same category and context (mirrors `BAT_SPEED_PREFERRED` in `batSpeed.ts`).
- `selectSpeedPicks(input)`: resolves a `SpeedTemplate` via `resolveSpeedTemplate`, then picks 1–5 movements from the catalog matching the template's required + optional categories, constrained by legality, season, training age, equipment, environment, and CNS/PAP budget.
- Stable day-of-week / seed rotation so athletes see variety but not randomness.

### 2. Replace `sprintSlugs` usage in the generator

File: `supabase/functions/wk-generate-daily/index.ts`

- Remove the `sprintSlugs(sport)` call and replace the speed block with `selectSpeedPicks(...)`.
- Keep the existing cadence gate (`hoursSinceSpeed >= block.speed_cadence_hours - 6`) and game-day logic.
- Push each selected movement into `rxs` with `slot="speed"`, `sequence_role="speed"`, and metadata (`speed_template_id`, `speed_required_category`) so `certifySpeed` can stamp it and diagnostics can record template/category coverage.
- Leave the current `certifySpeed` invocation unchanged; the richer selections will feed it naturally.

### 3. Adjust template/category mapping for existing data

The catalog currently has 8 speed categories; the canonical templates also reference `elastic`, `pap`, and `deceleration`. I will make the selector template-aware:

- Required categories that are missing from the catalog (e.g. `elastic` not yet seeded) will emit a diagnostics warning, and the selector will fall back to a related present category (`plyometric`/`reactive`/`mobility`) instead of failing certification.
- Optional categories that are missing will be skipped without error.
- This preserves the 80 existing movements while keeping the template vocabulary valid.

If the review confirms, I can also seed a small batch of `elastic`, `deceleration`, and `pap` speed movements to make every canonical template fully expressible, but the initial implementation works without waiting on new data.

### 4. Clean up legacy references

- `supabase/functions/_shared/wic/engines/sprint.ts` will become a thin re-export of the new `speed.ts` or be removed entirely, so nothing else imports dead code.
- Delete the import of `sprintSlugs` from `wk-generate-daily/index.ts`.

### 5. Frontend / legacy dailyPlan.ts (optional, only if still live)

If `HammerDailyPlan.tsx` still renders the legacy speed block from `src/lib/hammer/prescription/dailyPlan.ts`, I will replace its hardcoded `drillsMap` with a read of the backend Wk prescriptions (the same data the WkSpeedCard already uses) or a fallback that calls the catalog. The primary fix is the Wk cards, so this is secondary and only done if the component is reachable.

## Verification

1. Run the generator for a test user across `os_q1`, `os_q2`, `os_q3`, `os_q4`, `in_season`, `post_season`, `game_day`, and `practice_day` contexts and confirm each returns 2–5 speed prescriptions with distinct `speed_category` values.
2. Confirm `certifySpeed` passes (`validation_status = "passed"`) and no fatal `speed_unresolved_template` / `speed_duplicate_category` errors.
3. Confirm `wk_generation_diagnostics` records `speed_template_id`, `speed_category_coverage`, and `speed_validation_status`.
4. Confirm the WkSpeedCard in the preview renders movement names, sets/reps, and "Why this movement" for the selected prescriptions.
5. Confirm no speed movement is selected twice in one session and no movement is selected from outside the speed library.

## Rollout

- Deploy the `wk-generate-daily` edge function after edits.
- No database migration is required because the selector uses existing columns (`speed_category`, `season_legality`, `training_age_legality`, `equipment`, `pap_classification`, `movement_velocity`, etc.).
- If seeding new elastic/pap/deceleration movements is approved, I will run a small `supabase--migration` to insert them and then re-verify.
